01Visão Geral da Arquitetura

Aplicativo mobile híbrido em Flutter com backend serverless no Firebase. Sem servidor próprio — toda a lógica de negócio reside no cliente ou em regras declarativas do Firestore.

🖥
UI — Screens & Widgets
screens/ · widgets/ — sem lógica de negócio, apenas apresentação e interação
⚡
Estado — AppProvider (ChangeNotifier)
providers/ · router/ — único provider global; GoRouter ouve o provider para guards
⚙️
Serviços — FirestoreService · AuthService · WeatherService · LocationService
services/ — chamadas externas, transações, cache de sessão
📦
Modelos — AppUser · Atletica · Evento · ConfirmacaoUsuario · Local
models/ — entidades imutáveis com fromFirestore / toFirestore; utils/ para lógica pura
🔥
Backend — Firebase Auth · Cloud Firestore · FCM
Infraestrutura gerenciada: autenticação, banco de dados em tempo real, notificações push
ℹ️ Princípio de separação de camadas

Screens e widgets nunca fazem chamadas diretas ao Firestore nem executam cálculos de negócio. Eles leem estado via context.watch<AppProvider>() e delegam ações para AppProvider ou para métodos de FirestoreService injetados localmente. Toda lógica pura (viabilidade, ranking) vive em utils/viabilidade.dart e é testável sem Firebase.

03Sequência de Inicialização

O main.dart executa as seguintes etapas em ordem antes de exibir qualquer UI:

WidgetsFlutterBinding.ensureInitialized()
Garante que o binding Flutter esteja pronto antes de operações async.
dotenv.load(fileName: '.env')
Carrega as variáveis de ambiente, incluindo OPENWEATHERMAP_KEY. O arquivo .env fica nos assets mas não é commitado.
Firebase.initializeApp(options: DefaultFirebaseOptions.currentPlatform)
Inicializa o SDK Firebase com as credenciais da plataforma geradas pelo FlutterFire CLI em firebase_options.dart.
initializeDateFormatting('pt_BR')
Carrega dados de localização do pacote intl para formatação de datas em português.
AppProvider()..init() + buildRouter(provider)
Cria o provider global e o router. O router recebe o provider como refreshListenable — qualquer notifyListeners() no provider reavalia os guards de navegação.
MaterialApp.router com tema e ChangeNotifierProvider
O app renderiza com tema Material 3 (fonte Inter, paleta azul slate). O provider é injetado na raiz da árvore de widgets.

04Padrão de Estado — AppProvider

O AppProvider é o único ChangeNotifier global do app. Ele encapsula o estado de autenticação, o perfil do usuário logado e os dados da atlética vinculada.

CampoTipoDescrição

Ciclo de vida do estado de auth

void init() {
  // Escuta o stream do Firebase Auth — dispara sempre que o user muda
  _auth.authStateChanges.listen((user) async {
    if (user == null) {
      _usuario = null; _atletica = null;
      notifyListeners(); // → GoRouter redireciona para /login
      return;
    }
    await _carregarPerfil(user.uid); // busca users + atleticas no Firestore
  });
}

Telas leem o provider com context.watch<AppProvider>() para rebuilds reativos e com context.read<AppProvider>() para ações pontuais (ex: dentro de onPressed). Stores intermediários, Riverpod ou Bloc não são usados — o provider único é suficiente para a escala da V1.

05Roteamento — GoRouter

O router é declarativo e construído com go_router ^17. A instância é criada em main.dart e recebe o AppProvider como refreshListenable.

Guard de navegação

redirect: (context, state) {
  if (autenticado && carregando) return null; // aguarda perfil carregar

  if (!autenticado && path != '/login') return '/login';
  if (autenticado && !perfilCompleto && path != '/cadastro') return '/cadastro';
  if (autenticado && perfilCompleto && (path == '/login' || path == '/cadastro')) {
    return '/';
  }
  return null; // sem redirecionamento
},

AuthService

Atualização lazy de ranking

O pontuacaoRanking de cada usuário é recalculado no momento em que ele faz check-in ou recebe uma falta. A fórmula usa o maxComparecimentos da atlética consultado na hora, garantindo que a normalização seja sempre relativa ao melhor jogador atual.

⚠️ Desnormalização intencional

Os campos nomeUsuario e taxaConfiabilidade em ConfirmacaoUsuario são cópias do momento da confirmação. Isso evita N+1 reads ao exibir a lista de confirmados. 08Módulo de Cálculos — utils/viabilidade.dart

Toda lógica numérica está isolada em funções puras em utils/viabilidade.dart. Não há dependência de Flutter ou Firebase — testável com Dart puro.

Índice de Viabilidade

double calcularViabilidade({
  required int confirmados,
  required int minimo,
  required double chanceChuva,  // 0.0–100.0
  double? taxaFaltaMedia,       // null quando sem confirmados
}) {
  final fatorQuorum = minimo > 0 ? min(confirmados / minimo, 1.0) : 1.0;
  final fatorClima = 1.0 - (chanceChuva / 100);
  if (taxaFaltaMedia == null) {
    // Sem confirmados: confiabilidade excluída; máximo = 65%
    return (fatorQuorum * 0.20) + (fatorClima * 0.45);
  }
  final fatorConfiabilidade = 1.0 - taxaFaltaMedia;
  return (fatorQuorum * 0.20) + (fatorClima * 0.45) + (fatorConfiabilidade * 0.35);
}
FatorPesoCálculo
Quórum20%min(confirmados / minimo, 1.0) — nunca penaliza quórum intencionalmente baixo
Clima45%1.0 - (chanceChuva / 100) — campo pop da OpenWeatherMap
Confiabilidade35%1.0 - taxaFaltaMedia — média da taxa de falta dos confirmados

Pontuação de Ranking

double calcularPontuacaoRanking(
    double taxaConfiabilidade, int totalComparecimentos, int maxComparecimentos) {
  final fatorVolume = maxComparecimentos > 0
      ? totalComparecimentos / maxComparecimentos : 0.0;
  return (taxaConfiabilidade * 0.70) + (fatorVolume * 0.30);
}

A pontuação combina 70% qualidade (taxa de comparecimento histórico) com 30% volume (quantidade relativa ao melhor jogador da atlética). Atualização lazy: cada usuário recalcula sua pontuação ao fazer check-in ou receber uma falta.

Cobertura de testes

O arquivo test/widget_test.dart cobre os casos:

  • Condições ideais → viabilidade = 1.0
  • Quórum parcial (50%), clima 20%, confiabilidade ideal
  • Sem confirmados (taxaFaltaMedia == null) → máximo 65%
  • Classificação dos três níveis: Alta (≥70%), Méd. (40–69%), Baixa (<40%)

09Regras do Firestore (Security Rules)

As regras são declarativas e garantem que o cliente não possa ler ou gravar dados indevidamente. A autenticação é validada via request.auth.

rules_version = '2'; service cloud.firestore { match /databases/{database}/documents { // ── Usuários ────────────────────────────────────────────────── match /users/{userId} { // Leitura pública para autenticados (ranking, analytics) allow read: if request.auth != null; // Criação apenas pelo próprio usuário allow create: if request.auth != null && request.auth.uid == userId; // Atualização: próprio usuário edita perfil allow update: if request.auth != null && request.auth.uid == userId && request.resource.data.diff(resource.data) .affectedKeys().hasOnly([ 'nome', 'curso', 'atleticaId', 'role', 'taxaConfiabilidade', 'streakPresencas', 'totalComparecimentos', 'totalConfirmacoes', 'pontuacaoRanking' ]); } // ── Atléticas ───────────────────────────────────────────────── match /atleticas/{atleticaId} { allow read: if request.auth != null; // Apenas campos de gestão podem ser atualizados pelo cliente allow update: if request.auth != null && request.resource.data.diff(resource.data) .affectedKeys().hasOnly(['gestoresIds', 'faltasConsecutivas']); } // ── Eventos ─────────────────────────────────────────────────── match /eventos/{eventoId} { allow read: if request.auth != null; // Qualquer autenticado pode criar evento allow create: if request.auth != null; // Update: criador pode editar tudo; outros só campos de participação allow update: if request.auth != null && ( resource.data.criadoPorId == request.auth.uid || request.resource.data.diff(resource.data) .affectedKeys().hasOnly([ 'confirmados', 'indiceViabilidade', 'status', 'cientesCancelamento' ]) ); // Delete: apenas o criador allow delete: if request.auth != null && resource.data.criadoPorId == request.auth.uid; } } }
🔴 Ponto de atenção

O allow update em /users/{userId} inclui taxaConfiabilidade, streakPresencas, totalComparecimentos e pontuacaoRanking para permitir que o encerrarEvento() chame _registrarFalta() e realizarCheckIn() atualizem esses campos. Idealmente essas operações seriam feitas por Cloud Functions com Service Account — mas como não há servidor, a regra no cliente é a solução pragmática para a V1.

11Fluxos de Dados

Confirmar presença

  1. Usuário marca materiais e toca "Confirmar Presença"
  2. WeatherService verifica cache → se miss, chama API OpenWeatherMap
  3. calcularViabilidade() computa o novo índice com clima real e a taxa de falta média dos confirmados + novo confirmado
  4. Transação Firestore atômica: atualiza evento.confirmados[] + evento.indiceViabilidade + incrementa user.totalConfirmacoes
  5. Stream streamEvento(id) dispara → widget rebuilda automaticamente

Check-in por GPS

  1. Usuário toca "Fazer Check-in por GPS"
  2. Verifica janela de tempo (2h antes ↔ 6h depois)
  3. LocationService solicita permissão se necessário
  4. Obtém posição via GPS e calcula distância Haversine ao local do evento
  5. Se ≤ 50m: transação Firestore marca checkInRealizado = true + timestampCheckIn
  6. Após transação: _atualizarConfiabilidadeUsuario() incrementa totalComparecimentos, streakPresencas, recalcula taxaConfiabilidade
  7. Atualização lazy do ranking: busca maxComparecimentos na atlética → grava pontuacaoRanking

Encerramento de evento (gestor)

  1. Gestor toca "Encerrar Evento" (visível apenas após dataHora passada)
  2. encerrarEvento(eventoId): muda status → encerrado
  3. Para cada confirmado sem check-in: _registrarFalta(userId) — zera streakPresencas, recalcula taxa
  4. AlertDialog de resumo exibe: total confirmados, total check-ins, nomes de quem faltou

Recálculo de índice na abertura do detalhe

  1. _carregarClima(evento) é chamado via addPostFrameCallback
  2. WeatherService retorna chanceChuva atual
  3. Se o novo índice difere > 1% do armazenado → atualizarEvento(id, {indiceViabilidade})
  4. Isso garante que o índice reflita o clima atual sem re-fetch desnecessário
